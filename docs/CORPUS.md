# CORPUS: mnema 目标概念模型（v2）

状态：目标模型，非当前实现。当前代码结构见 ARCHITECTURE.md。
与现实现的差异集中在一次 schema 迁移中处理（第 10 节）。
决策过程记录于 journal 设计线（2026-07）。

## 1. 设计立场

场景：LLM agent 承担跨会话的长期工作。进程随时可能终止，后继
进程没有记忆，journal 是向后继传递工作历史的唯一介质。

职责划分只有一条：**机器负责记录与检索，不做语义解释；理解由
读取时在场的 LLM 完成。**

机器的合法操作限于两类：

- 元数据记录：时间戳、顺序、写入通道、内容完整性；
- 检索原语：分组、排序、过滤、指针跟随、计数。

"这段话什么意思""这个结论今天还成立吗"——由读取时的 LLM 回答，
机器不预置答案。

由此得到读写契约：**每一种结构化读取路径，对应写入时的一个
结构化承诺。** 机器不理解内容，无法事后补建结构——写入时未标注
[decision]，之后就无法按决策过滤，只能全文重读。

## 2. 实体：entry

系统唯一的对象是 entry（一条记录）。线（strand）是 entry 的链；
link 是带特定 effect 的 entry；ref 是 entry 的一个字段。不存在
独立的"边"对象（第 4 节）。

entry 由九个部分组成，分四层：

| 层 | 字段 | 来源 | 说明 |
|---|---|---|---|
| 元数据 | 时间戳 | 机器 | 写入时刻 |
| | 前驱 | 机器 | 本线上一条的 id，构成线内顺序 |
| | offset | 机器 | 在本 journal 中的落盘序号（局部索引） |
| | 作者 | 机器+声明 | 机器可证明的仅到通道（进程/会话）；更具体的身份由作者声明，两部分分开存储 |
| | id | 推导 | 以上与以下全部内容的哈希（第 3 节） |
| 结构承诺 | marker | 作者 | 条目类型标签，一个词，如 [decision]（第 5 节） |
| | refs | 作者 | 本条所依据的旧记录（第 4 节） |
| 内容 | 正文 | 作者 | 自由文本，面向未来读者，标准见下 |
| 状态变更 | effect | 作者 | 是否改变线的状态，默认 none（第 5 节） |

写入成本不因此增加：元数据全自动，marker 一个词，refs 与
effect 可选。

正文书写约定（非字段）：需要按轨迹提取的数值与状态用
`名称=值` 格式，如 `[metric] tests_passed=236`。

**正文的自包含标准：单位是线，不是条。** 读者模型：按顺序读过
本线在前内容、但没有写入者会话记忆的 LLM。规则四条：

1. 线内在前内容可直接引用，不必复述；引用距离越远越需要显式
   把手——相邻条目可写"上一条"，间隔数十条应给 ref 或简述
   （实际读者常只读末尾若干条）；
2. 线外依赖必须给可解析的地址：其他线用 ref，文件给路径，
   代码给 commit；
3. 会话中的临时名称不得直接使用，判据："该词在本线在前内容中
   有定义吗"；
4. 首条 entry 没有在前内容，须完全自包含——它是读者决定
   是否读这条线的依据。

## 3. 身份：哈希链与内容寻址

每条 entry 的 id = hash(前驱 id ‖ 本条内容)。这是内容寻址
（与 git 对象模型相同）：身份由内容决定，与存储位置无关。

链式哈希的直接后果：修改历史上任何一条，其后全部 id 失效——
防篡改从依赖纪律变为可校验（doctor 顺链验证即可定位断点）。

**完整性靠结构，不靠检查。** 保证一条规矩成立有两种办法：事后
派检查器查（允许违规先发生、再报出来），或让违规在结构上无法
存在（无需检查）。哈希链把一批完整性规矩从前者挪到后者：改坏
任何一条记录会使其后全部 id 失效，"记录有没有被改"验一遍链即可
查出，不再需要单独的损坏检查；顺序焊在 id 里（每条 id 含上一条），
重排即断链，顺序不可能乱；线的 id 就是首条的哈希，身份是定义，
不可能不匹配。因此**数据模型自带的完整性越强，doctor 事后要查的越少**。
但这只对完整性这一类成立（顺序、身份、防篡改本就该不可能违反）；
像 ref 失效、belongs-to 指向已关母线这种，是需要被看见的事实，
不是需要被杜绝的缺陷，无法也不该结构化掉，永远保留。

链按线建立，不建全局链。全局链把"不可合并"写入密码学（两机
分叉后无法归一），并破坏线的可移植性；按线建链，每条线携带
自身的顺序证明，可被跨 journal 引用。

历史不可重排，除了哈希把顺序焊死，还有更朴素的一层：一条 entry
是接着上一条写的，搬到别处上文就变了、话也跟着变味——所以重排一条
记录不只是被哈希拦住，是本来就没意义。这也是第 7 节"别人的记录塞
不进你的 journal、只能被你重新说一遍"的根。

**线的 id 即其首条 entry 的 id。** 线不是先建好再装入记录的
容器，而是从首条记录延伸的链。

offset 降级保留。它只在本 journal 内有意义（类似页码），保留
两个用途：本地快速定位；跨线交错顺序（全局时间线只能来自落盘
顺序）。不再充当引用地址与线内顺序证据——两者由内容哈希与链
承担。

**journal 级完整性：锚点事件。** 按线建链有三个盲区：整线删除
不可检测（其余链各自完好）；跨线交错顺序无保护；两份副本缺少
快速比对手段。对策：机器周期性写入锚点事件，内容为当前全部
线头哈希的清单及其摘要（类似 certificate transparency 的
signed tree head）。doctor 逐个锚点核对重放结果——删线或回退
即暴露；相邻锚点构成粗粒度顺序窗口；最新锚点的摘要即整个
journal 的状态校验和，副本比对只需比较一个值。锚点内容纯为
哈希运算，机器自动写入。

已知边界：链与锚点防"改历史"，不防"全量重算"（有写权限者可
重算全部哈希）。对策是外部锚定：journal 纳入 git 后每次 commit
即一次外部存证。三层分工：链保证线内不可改；锚点保证线不可删、
副本可比对；外部锚定防全量重算。

## 4. 关系：link 与 ref

两种关系机制，均不引入独立的边对象。**link 表达线与线的关系，
供机器计算投影；ref 表达记录与记录的依据，供读者追溯。**

**link。** 两种类型：

- belongs-to：子线属于母线，归属唯一，递归构成树（第 6 节）；
- depends-on：本线等待目标线先完成。

关系类型进入 link 的准入条件是**存在消费它的投影**：tree 命令
读全部 belongs-to 边渲染嵌套层级（谁是谁的一部分，每节点带
生命周期状态）；depends 命令读 depends-on 边并结合各线的关闭
状态，计算就绪与否、阻塞在哪条线、未完成依赖的最长链。无投影
消费的关系（merged-into、supersedes 等）只需被读者跟随、不需
被机器计算，用 ref 加 marker 表达，不设 link 类型。

link 的实现是一条 effect 为 belongs-to/depends-on 的 entry，
写在声明方（源线）的链上，因而自动具备时间戳、作者、正文
（建边理由）。

撤销采用冲销而非删除：写一条 unlink entry，按 id 指明冲销
对象。两条均永久保留，仅在计算当前状态时相互抵消。

单侧存储、双侧可见：仅存于源线；目标线视图经反向查询获得入边。

**ref。** 就像论文里的引用、或聊天里的"引用回复"：写新记录时
附上所依据旧记录的地址。示例：在线 A 写"[decision] 采用方案 B，
放弃方案 C"，依据是实验线中"C 在真实数据上内存溢出"的记录——
写入时加 `--why <该记录>`，该记录的哈希即存入本条。

ref 在三个时点起作用：

- 写入：一个旗标；
- 读取：读者按哈希直接取回被引记录原文。没有 ref 时依据只是
  散文（"如之前实验所示"），读者只能全文检索并猜测；
- 失效检测：被引记录所在线越过引用时点后，doctor 提示"依据有
  后续更新"。doctor 只报告位置事实，是否推翻结论由 LLM 判断。

规则三条：

1. 只能为当前正在写的条目添加 ref。事后补充"旧记录其实基于 X"
   应写为新记录；
2. ref 参与本条哈希，写入后不可修改；
3. 推翻结论写新记录（[correction] + ref 指向被推翻条目），
   不修改历史。

地址为被引 entry 的哈希，跨 journal 有效。机器对 ref 仅执行
存储、解析、反向查询、位置比较，不读内容。

## 5. marker 与 effect

同一 entry 上的两个标签字段，受众相反。

**marker 面向未来的 LLM。** 开放词表，作者可自由扩展；机器
视为不透明字符串，仅存储与过滤。

实际使用沉淀的惯用词表（约定，非强制）：

| marker | 含义 |
|---|---|
| [decision] | 决策 |
| [observed] | 客观观测，不含判断 |
| [friction] | 使用中遇到的摩擦 |
| [correction] | 推翻或修正在前条目 |
| [deliverable] | 产出物 |
| [pause] | 不可逆动作前的自省留痕 |
| [metric] | 数值，配 名称=值 |
| [context] | 背景信息 |

工具对疑似拼写偏差给出提示（编辑距离），不拦截。

**effect 面向机器。** 封闭词表，机器必须理解以计算状态：
none（默认）、close(处置)、link(类型,目标)、unlink(目标id)、
hide、unhide、reopen。

规则三条：

1. 准入测试：新 effect 必须回答"哪个投影因它改变输出"。答不出
   则它是 marker（例：[metric] 无投影消费，不进 effect）；
2. 带 effect 的条目，marker 由 effect 推导（渲染为
   [closed(done)] 等规范形式），不可另填，防止两字段矛盾；
   作者的说明写入正文；
3. 状态只由 effect 改变，marker 绝不改变状态。marker 至多用作
   显示层提示（如末条是 [done] 时建议运行 close）。两字段分立
   正是为此：一个供人读、可自由取值，一个供机器算状态、必须
   受控；混用会让随手写下的一个词意外改动线的状态。

## 6. 折叠：靠生命周期，不靠摘要

问题：历史增长，冷启动读取成本随之增长。

**否决的方案：摘要文档**（"当前净状态"类总结）。理由：新记录
写入即过期；构成与原始记录并存的第二真相源；将前任作者的理解
预置给后任读者，妨碍独立重审。

**采用的方案：**

- close 是带 effect 的 entry，携带理由，写于事项了结之时——
  作者理解最完整、且结论不再变动的时刻。一条线的 close entry
  即其结题记录。
- 活跃视图 = 过滤已 close 的线。机器仅过滤，不概括。
- **reopen 仅表示冲销错误关闭**（误关、验收未过即关），与
  unlink 对称，须附理由，原线继续。判据："关错了，还是世界
  变了？"世界变化（缺陷回归、需求恢复）不用 reopen——那是
  新事项：正常开线，首条 ref 旧线的结题记录；旧线保持关闭，
  其结题在其时点上仍然成立。这不是新机制，是 add 与 ref 的
  常规用法——一个事项的完整历史是一串短线，不是一条无限
  增长的线。

**拆分子线**，判据一句话：

> 这一段是否可能在某天被单独宣布完成？
> 是——独立事项，拆分；否——母线的情节，保留。

判据须在写入时刻可执行，不依赖事后视角。事前判据，写入前可问
的三个测试：本条能否说出独立于母线的完成标准？本条是否为工作
计划（计划自带完成标准）？本条能否分派给其他 agent 独立执行？
任一为是，开线。

无法说出完成标准的暂不拆分，由事后症状兜底：话题交错迫使
跳读；需要"线内结构"；外部反复引用本线的某一局部；该段接近
完成而母线未完。拆晚代价有限（历史留在母线，ref 可补引用），
事前判据允许保守。

**粒度的两端是同一类错误的两个方向。** 过长 = 事项内含未拆分
的事项；过碎 = entry 被当作事项（一两条即完结的内容占用活跃
列表名额、附带 close 义务与导航跳转）。长度只是症状，病因都是
对"独立起止"的误判。默认策略：小内容先写为 entry，出现后续再
升格；一旦能说出完成标准，立即拆分。

拆分动作（append-only 下历史不可移动，拆分只影响未来）写两笔，
均存于子侧、指向父侧（父侧视图由反向查询生成），两笔是两个
不同的关系而非一个关系的两个方向：

- **归属（belongs-to link）**：指向母线，表示当前的结构归属；
  可变更（冲销后重建）；
- **来源（子线首条的 ref）**：指向母线中的具体记录（粒度是
  记录而非线），表示本事项由哪些判断产生；不可变更。

归属与来源正交，三种组合均实际存在：归属变更而来源不变；
有来源无归属（从其他线的讨论中产生但组织上独立）；有归属无
来源（计划内工作直接挂于母线）。

母线中的边界记录（"X 转入子线"）为可选：不写时投影可经反向
查询标出派生点；写时母线主干阅读更连贯。工具在派生时提示，
不强制。

派生应为原子命令：`add --parent <母线> --from <来源记录>`
一次写入子线首条、来源 ref、belongs-to。派生成本低，长线才会
实际被拆——纪律依赖成本结构，不能只依赖自觉。

旧记录留在母线原地，因此拆早优于拆晚。

冷启动阅读由此成为分层下钻，全程无摘要文档：活跃线列表
（已关线整批退出视图）→ 母线主干（决策、边界记录、整合结论）
→ 按需进入子线全文；已关子线只读结题记录。

## 7. 边界：journal

- **一个 journal 对应一个项目。** 发现方式与 .git 相同（自当前
  目录向上查找）。记录密集引用项目文件与 commit，分离存储即
  失去自包含性。
- **单写者。** 同一 journal 同一时期只在一台机器写入（单机并发
  由锁处理）。不实现多机同步协议。
- **没有主 journal。** 一份 journal 属于动手实践、一步步把它记
  下来的那个人——所属跟着人走，不跟着项目走。fork 一个仓库，代码
  同源，journal 却分家：动手的人分开了，各记各的，往后不再有交集，
  也不需要合并。别人的 journal 你能读（当参考）、能按哈希引用其中
  一条，但两件事不行——不能接着它往下写（把自己的话塞进别人的历史，
  等于假装两个人是一个人），也不能几个人合写一份再 merge（两条各自
  长出来的历史拼不回一条，同多机分叉，第 3 节）。外面的想法要进你
  的 journal 只有一条路：你读懂、认下，用自己的话记一条，注明来源。
- **明文 JSONL。** 工具二进制缺失时，cat/grep/jq 仍可读取全部
  历史。记录的存活期必须长于工具。
- **跨 journal 引用。** journal 身份 = 首条事件的哈希。跨
  journal ref = journal 身份 + entry 哈希。联邦查询（只读）：
  本地注册表记录已知 journal，时间线可跨库归并——库内顺序
  精确，库间按时间戳近似，如实标注。
- **纳入 git。** 单机场景收益明确：clone 即备份，历史随仓库
  迁移，并为哈希提供外部锚定（第 3 节）。多机场景以内容寻址
  落地为前提（否则合并时序号冲突，既有引用失效）。可见性须
  显式决策：journal 与代码同级公开，仓库开源即 journal 公开。

## 8. 通道与运行时约定

**写入统一为 stdin 通道。** 正文不经命令行参数传递——参数经过
shell 解析，引号、$、以连字符开头的内容均可能出错；stdin 内容
原样到达。分工：旗标携带结构（--id、--why、effect 参数），
stdin 携带正文。stdin 为终端且无管道时立即报错，不阻塞。

**人类引用薄壳。** 机器层引用仍以内容哈希为正典：journal 里存
全宽 id，JSON 输出的 `id`/`strand_id` 字段保持全宽哈希，供脚本、
provenance、`--why`/ref 等稳定 join。人类操作的痛点是回忆与转录，
不是缺一个独立交互界面；因此只在 CLI 入口上加可丢弃的解析便利，
不得改变机器契约。

人类入口可接受四类把手：

- 唯一短前缀：像 git 一样，任意长度前缀只有唯一匹配时才解析；
  歧义必须报候选，不得选第一条；
- slug：创建线时给出的持久别名，随 journal 保存，供人跨会话使用；
  slug 与哈希前缀机械分离（禁纯 hex），机器 join 仍用全宽 id；
- `@last` 与 `@1`/`@2`：CLI 本地选择缓存，分别指最近触达线与上次
  `list` 文本输出的序号；缓存只减少转录成本，不是事实源；它是
  journal 目录级共享状态，不按进程或 agent 隔离，可能被并发写者覆盖；
- `pick`：交互式选线。有 `fzf` 用 `fzf`（带预览窗格，光标划过即看该线
  头尾），无 `fzf` 退回内置箭头菜单（↑/↓ 移动、打字过滤、Enter 选中）。
  `pick` 也能写：选线走终端、正文走 stdin（`… | mnema pick append`），
  正文永不从终端读。

失效语义必须偏保守：重复 slug 或异常冲突时报歧义；`@1`/`@2` 在
journal 前进、映射缺失、状态文件损坏或候选不存在时明确报错并提示
重跑 `list`；选择缓存只在文本模式读写，`--format json` 不产生缓存
副作用；非 TTY 下任何交互选择入口不得挂起等待输入。交互模式也不得
成为任何能力的唯一入口，同一操作必须始终能用显式参数完成。

**同一份事实，两个投影。** 底层 entry 只有一份；机器与人各自投影它，
读法相反。机器无跨进程记忆、按 token 计费、无终端，读法是批量吞入、
系统重建，要的是完整与机器可导航结构——它的投影是 `--format json`。人
有会话连续性但注意力有限，读法是扫、认、早弃，多数时候只是再定位，要
的是最小面加按需展开——它的投影是折叠、渐进、可交互的文本形态。`pick`
就是人投影的交互形态，对应下文「分层读取」的「首尾」层（每行给出「是
什么／到哪了」），选中即向下钻一层（`show --tail`）。两个投影只在呈现
与交互层分叉，读的是同一份 projection，不建第二真相源。

人侧写入沿用同一条纪律：**交互只用来选线，正文永远走 stdin。** `pick
append` 选线交互、正文管道，没有管道正文即报错，绝不从终端读正文——
那会既破「写入统一为 stdin」（本节首），又诱使写出只在会话内成立的碎
条（§2 自包含标准）。交互属于人这一侧的便利，写入的事实质量线两侧同守。

**查询分工：工具收窄范围，jq 过滤字段。** 一件事归谁做，五条：

| 这件事 | 归谁 | 理由 |
|---|---|---|
| 计算投影，或工具约定的规范化 | 工具 | jq 重新实现投影语义，每个消费者各错一版 |
| 传递闭包 / 图遍历（子树展开、链遍历） | 工具 | jq 不适合递归 |
| 按哈希随机定位 | 工具 | 工具有索引，jq 只能全量扫描 |
| 字段值匹配 | jq | 放进工具就是查询语言的开端 |
| 字段选择、分组、聚合 | jq | 塑形是 jq 的职责 |

工具据此提供：全部投影（orient/tree/depends/timeline/digest）、
范围参数（--tail/--head/--range/--since-offset/--under）、哈希
解析、链与锚点校验、规范化字段的完整暴露（marker_head、refs
——工具少暴露一个规范化字段，jq 侧即多一批各自实现的错误
版本）。边界立场：ref 反向查询先由 jq 承担，journal 达万条级
后工具原生化（与 --backlinks 对称）；跨 journal 联邦查询只能
由工具承担。工具不提供复合查询语言。典型组合：

```bash
mnema timeline --under <母线> --since-offset 400 --format json \
  | jq '[.events[] | select(.marker_head == "decision")]'
```

**时间：存储与显示分离。** 存储绝对时间戳（机器写入）。显示用
相对加绝对（"3天前 (06-29)"——纯相对表述在长对话中过期，纯
绝对时间戳对无时间感的 LLM 无意义）；线内长间隔由机器直接标注
（"距上条 19 天"），不留给读者计算。顺序的依据是链，不是时钟
——时钟仅用于与外部世界（git log、发布日期）对齐。

**线的分层读取。** 不默认全量读取。六种读取方式全部由位置与
标签操作提供，选择权在读者：

| 方式 | 机器操作 | 用途 | 成本 |
|---|---|---|---|
| 首尾 | 首条+末几条 | 判断这条线是什么、进展到哪 | O(1) |
| 尾部 | --tail N | 接续工作（依赖正文标准的就近引用规则） | O(N) |
| 主干 | 按 marker/effect 过滤 | 只读决策、修正、边界、结题 | O(结构点) |
| 增量 | --since-offset | 上次读取之后的新增 | O(增量) |
| 指针 | 沿 refs 跳转 | 依据追溯与影响分析，深度由读者控制 | O(所引) |
| 全量 | 顺链全读 | 审计、重审、撰写结题——确需全部记录 | O(全部) |

升级顺序：首尾 → 尾部或主干 → 指针 → 全量；每层读完再决定
是否进入下层。

超长线的三层对策：预防（过长即拆分信号，第 6 节）；主干读取
将必读集从全量压缩到结构点（先 --digest 查看 marker 分布再
选取）；确需全量时按 offset 区间分片迭代读取，读者在自身上下
文中维护工作摘要——该摘要用后即弃、不写回，与摘要文档的否决
（第 6 节）不冲突：否决的是持久化的特权摘要，不是读者的草稿。

**冷启动仪表盘：orient。** LLM 每次启动只调用一次 orient，之后改用
上面的分层读取，不再回到 orient。所以 orient 要在这一次里给全冷启动
需要的东西，不追求简短——简短只对反复读的内容才有意义。

orient 自己不产生数据，它把几个视图拼在一起，回答冷启动时的四个问题：

- **我在哪**：journal 的身份、完整性（锚点是否校验通过）、规模；
- **有哪些事在进行、我接哪件**：正在活跃的线，母线连带子线的状况；
  每条线显示首条（这是什么）和末条（进展到哪），并给出短前缀的
  跳转命令；
- **有什么要我判断**：看起来做完却没 close 的线、完整性告警。只放
  读者知道后会改变下一步的事实（判断线间进展漂移需要基准，不放进
  默认，用 orient --since）；
- **怎么动手**：常用命令清单（add / append / close / show / tree /
  timeline / search / explain），加上 pause 自省全文。

放不放全文，看这份内容在别处有没有更该出现的地方。写入文法的细节，
真要写时 append --help 就在手边；完整性细节随时能跑 verify；某条线的
细节用 show 或 tree 展开——这些在 orient 里只留一句话或一个入口。
唯独 pause 没有别的落点：工具拦不住动作发生的那一刻，冷启动的 LLM
也不会主动去查它，所以它的全文只能放在 orient。pause 是纪律不是关卡，
工具只让"动手前停一下"留下痕迹，不阻止动作本身。

orient 里的每块内容，只要在冷启动之外还有人会单独查，它就同时是一条
独立命令，orient 里只留精简版——比如"要我判断的"这块，就是 doctor
检查在冷启动时的一次呈现，同一份计算、两个出口。只有像 pause 这种
仅在冷启动才用得上的，才由 orient 独占。

**读取视图是九个字段的投影，不是九个字段本身。** 九个字段是
存储用的完整 schema，面向机器校验与审计；LLM 读取时几乎从不
需要全部九个，读了也是浪费。默认视图只呈现承载意义的字段，且
都翻成给人看的形式：

- **时间**：相对加绝对（"3天前(06-29)"），长间隔直接标出（本节
  时间约定），不显示原始高精度时间戳；
- **marker**：直接显示；带 effect 的条目显示为 effect 推导的
  规范形式（如 [closed(done)]），effect 为 none 时不单列；
- **正文**：完整显示；
- **refs**：仅在有时显示，渲染为短哈希加取回命令，不显示完整
  目标哈希；
- **短 id**：约七位，作为跳转与引用的把手；
- **作者**：仅当这份 journal 有多个写入者时才显示——单写者时
  每条都相同，显示即噪音。

其余字段默认不向 LLM 显示，因为它们只服务机器：前驱哈希（链的
内部机制，顺序已由阅读顺序体现）、完整长度的哈希、作者的通道
细节。offset 作为位置把手，只在需要定位的视图（orient、timeline）
中出现，不逐条重复。

这些都在完整视图（--format json，全部九个字段）里，供审计与 jq
使用；LLM 只在要用 jq 塑形或做审计时才降到完整视图。省略只发生
在显示层，存储永远完整。

**诊断的命名。** 每一类诊断用可读的名字标识（如 stale-ref、
ref-target-advanced），名字本身说明是什么事实，读者无需查表即懂，
机器也能按名字匹配（退出码、JSON 字段）。诊断在与该事实相关的
读取处就地出现，各附一条"去重新查看"的命令；看到之后是否据此
改变结论，由 LLM 判断，工具不代为决定。

## 9. 自检：doctor

doctor 不是独立的子系统。它和 tree、depends 一样，是 journal 对
自己的一次投影，读的是 journal 自身的结构。它做两类事，性质不同。

**一，验证完整性——可以判定成败。** 顺着哈希链与锚点重算一遍，
回答一个客观问题：这份 journal 有没有被改过、删过、乱过。这类
检查有明确对错，可给出退出码供 CI 使用。数据模型自带的完整性
越强，这里要验的越少——哈希链把顺序、身份、防篡改都焊进了 id，
doctor 只需验链，不必再单独查这些（第 3 节）。

**二，把值得注意的事实摆出来——永远不判成败。** 例如某条依据
在被引用之后又更新了、某条线归属于一条已关闭的母线。这些不是
错误，是事实：机器用跟指针、比位置就能查出，但它们要不要改变
做法，只有读取时的 LLM 能判断。

这类事实大多不该攒成一份报告，而应在相关的读取处就地出现——
失效的依据在你 show 那条记录时标出，归属已关母线在 tree 里标出。
道理和写入时的提示一样：事实要出现在读者正站着的地方，而不是
躲在一份要专门去跑的报告里。只有那些没有自然去处的事实（例如
很久没动、又一直没关的线），才由 doctor 单独列出。

摆什么、不摆什么有一条纪律：能查出来是一回事，该不该摆是另一
回事。判据是"读者知道了会不会改变下一步"。一条没人会再读的
已关线上的失效依据，查得出，摆出来只是噪音；一条你正要依赖的
记录失效了，才值得摆。

doctor 从不做这几件事：

- 不判对错。只说"变了"，不说"错了"。
- 不留状态。输出一次性，重算即得，不写回 journal，也没有"这条
  我确认过、别再提"的记号——一个事实成立到底层记录变化为止，
  不能被标记掉。
- 不排轻重。它列举，轻重由读者定。
- 不动手修。它可以给一条命令让你去看，但从不自己执行；给命令与
  执行命令之间那道缝，正是留给判断的。它给的命令永远是"去重新
  看"，不是"照此修复"——说"这样修"就等于替你判定了它该被修。
- 不拦工作。除完整性那类给 CI 的退出码外，doctor 从不阻止你
  继续；一份被改坏的 journal 你照样能读，只是会知道哪部分不可信。
- 不裁量它算不出的判断。它能数出一条线有多少记录，但不会说
  "太长了，该拆"——数量是事实，"太长"是判断，判断是 LLM 的
  事（第 6 节）。

doctor 与 LLM 的分工正是整个模型的缩影：doctor 把事实摆出来
（一次性），LLM 判断（"这条失效的依据确实推翻了我的结论"），
判断才写回成新记录（[correction] 加 ref）。doctor 是证人，写
判决的是 LLM。

两个诚实的边界：跨 journal 的依据本地验不了，doctor 只报"这条
指向别处，本地无法核实"，不假装验过（第 7 节）；哈希链与锚点防
改历史，但防不住有写权限者把整份重算，这一层只能靠纳入 git 由
外部留存（第 3 节），doctor 验到最新锚点为止，越过即如实声明
无能为力。

## 10. 迁移

本模型与当前实现的差异，集中在一次 schema 迁移中处理：

1. id 体系：随机 id → 按线哈希链，线 id = 首条哈希；
2. 引用地址：线id@offset → entry 哈希（过渡期双存）；
3. 事件类型：StrandClosed 等独立类型 → 统一为带 effect 的 entry；
4. 写入通道：位置参数/--stdin/--file → stdin 单通道；
5. link 存储：独立边事件 → 源线链上的 effect entry；
6. journal 完整性：新增锚点事件与 doctor 锚点校验。

迁移前的 v1 改进（--stdin 教学优先、回执提示、marker_head
字段、管道 panic 修复等）不破坏现有约定，见 journal 工单线。
